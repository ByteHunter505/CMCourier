## 1. Approach Summary

A single class `TabularDataSource` at `src/cmcourier/adapters/sources/tabular.py` implements `IDataSource` over CSV and XLSX files using pandas. File format is dispatched by extension at construction time; the adapter holds an in-memory `DataFrame` for the lifetime of the instance.

The implementation is intentionally small (~120 lines). All real work is delegated to pandas; the adapter's job is dispatch, normalization (`NaN` → `None`), and clean error wrapping at the port boundary.

## 3. Architectural Decisions

### 3.1 One class, two formats

A single `TabularDataSource` handles both CSV and XLSX rather than two sibling classes (`CSVDataSource`, `XLSXDataSource`). Reasons:

- The IDataSource methods are identical — only file loading differs. Two classes would duplicate ~80% of the code.
- File format is a property of the input, not of the contract. Callers don't care which format the source is.
- If a third tabular format appears (Parquet, JSON Lines), it joins the dispatch with minimal code change.

### 3.2 `query()` raises `NotImplementedError`

The IDataSource port has `query(sql, params)` and `query_stream(sql, params)` methods. The AS400 adapter (later change) implements them via `pyodbc.cursor.execute`. The CSV/XLSX adapter has no SQL surface — there is no engine to execute the query against.

Three options were considered:

1. **Implement SQL via `pandasql` or `duckdb`** — both work, both are heavy dependencies, both have surprising semantics on edge cases (NULL handling, JOIN behavior). Adding a SQL parser to fake a feature the data source doesn't natively have is a tar pit.
2. **Split the IDataSource port into `IFieldSource` + `ISqlSource`** — Interface Segregation Principle. The "right" answer architecturally, but it requires amending the spec of 002 and re-authoring the port hierarchy. Not justified for one method on one class.
3. **Raise `NotImplementedError`** — Python idiom for "method exists per the contract but this implementation does not support it". Callers that need SQL must use the AS400 adapter; the service layer that wires adapters knows which one is in play.

Option 3 chosen. The error message is explicit:

```python
raise NotImplementedError(
    "TabularDataSource does not support raw SQL. "
    "Use get_by_fields(filters), get_by_fields_in(field, values, ...), or get_all() instead."
)
```

If the project later splits the port (per option 2), `query()` simply becomes part of an interface that `TabularDataSource` does not implement. Migration is mechanical.

### 3.3 Eager load, in-memory `DataFrame`

`pandas.read_csv` and `pandas.read_excel` are called once at construction. The full file lives in memory until `close()`.

Justifications:
- **Test fixtures are tiny** (single-digit MB at most). Memory is not a constraint.
- **`get_by_fields` / `get_by_fields_in` are O(N) per call**. Building indexes is premature optimization.
- **Streaming via `chunksize`** would complicate `count()` and require a different code path for `get_by_fields` (multi-pass over chunks). Defer until a real-world workload demands it; document under post-MVP roadmap when this change merges.

### 3.4 `dtype=str` always

Both `read_csv` and `read_excel` are called with `dtype=str`. Reasons:

- Preserves leading zeros (`"000456"` stays `"000456"`, never becomes integer 456).
- Eliminates pandas's silent type inference, which differs subtly between CSV and XLSX for the same input.
- The service layer (004+) is the right place to convert strings into typed domain values via factories like `RVABREPDocument.from_row(...)` or `parse_cymmdd(row["creation_date_str"])`.
- Adapters do **transport**, not **interpretation**.

This is non-negotiable in this change. No `dtype` parameter is exposed.

### 3.5 `NaN` → `None` normalization

`pandas` represents missing values as `numpy.nan` (a float) regardless of the column's declared type. Yielding `nan` to callers leaks pandas-specific semantics through the port boundary and breaks `is None` checks.

Implementation: a small helper at module scope:

```python
def _normalize_row(row: dict[str, Any]) -> dict[str, Any]:
    return {k: (None if pd.isna(v) else v) for k, v in row.items()}
```

Applied to every dict yielded from `get_all`, `get_by_fields`, `get_by_fields_in`, `query_stream`. Cost is per-cell, but the workloads are small enough that this never lands on the hot path of a real migration.

### 3.6 `Path` only, no `str`

Constructor accepts `pathlib.Path`. A `str` would be convenient but encourages bug-prone string concatenation at call sites. PEP 519 path objects are the modern Python convention.

### 3.7 Encoding default UTF-8

`encoding="utf-8"` is the constructor default. Real-world RVI exports may surface in Latin-1 or Windows-1252; callers override per source. No auto-detection (chardet) — it's a post-MVP item if it becomes painful.

### 3.8 Sheet selection for XLSX

XLSX files may have multiple sheets. The constructor accepts `sheet_name: str | int = 0` matching pandas's parameter. Default is sheet 0 (first sheet) which is the common case. Multi-sheet sources select explicitly.

`sheet_name=None` (which would tell pandas "read all sheets") is **rejected** at construction with `ConfigurationError`. The adapter is "one source = one sheet". If a caller needs a multi-sheet aggregation, they generate it ahead of time or write a wrapper.

## 4. Implementation Sketch

```python
"""TabularDataSource — concrete IDataSource over CSV/XLSX files."""
from __future__ import annotations

__all__ = ["TabularDataSource"]

from collections.abc import Iterator, Mapping
from pathlib import Path
from typing import Any

import pandas as pd

from cmcourier.domain.exceptions import ConfigurationError
from cmcourier.domain.ports import IDataSource

_SUPPORTED_CSV = (".csv",)
_SUPPORTED_XLSX = (".xlsx", ".xls")


def _normalize_row(row: dict[str, Any]) -> dict[str, Any]:
    return {k: (None if pd.isna(v) else v) for k, v in row.items()}


class TabularDataSource(IDataSource):
    """In-memory IDataSource over a single CSV or XLSX file. See spec.md / plan.md."""

    def __init__(
        self,
        path: Path,
        encoding: str = "utf-8",
        sheet_name: str | int = 0,
    ) -> None:
        if not path.exists():
            raise FileNotFoundError(path)
        suffix = path.suffix.lower()
        if suffix in _SUPPORTED_CSV:
            self._df = self._load_csv(path, encoding)
        elif suffix in _SUPPORTED_XLSX:
            self._df = self._load_xlsx(path, sheet_name)
        else:
            raise ConfigurationError(
                f"Unsupported file extension {suffix!r}; expected one of "
                f"{_SUPPORTED_CSV + _SUPPORTED_XLSX}"
            )
        self._closed = False
        self._path = path

    @staticmethod
    def _load_csv(path: Path, encoding: str) -> pd.DataFrame:
        try:
            return pd.read_csv(path, encoding=encoding, dtype=str, keep_default_na=True)
        except (UnicodeDecodeError, pd.errors.ParserError, pd.errors.EmptyDataError) as exc:
            raise ConfigurationError(f"Failed to load CSV {path}: {exc}") from exc

    @staticmethod
    def _load_xlsx(path: Path, sheet_name: str | int) -> pd.DataFrame:
        try:
            return pd.read_excel(path, sheet_name=sheet_name, dtype=str, engine="openpyxl")
        except Exception as exc:  # pandas / openpyxl exceptions vary
            raise ConfigurationError(f"Failed to load XLSX {path}: {exc}") from exc

    def _ensure_open(self) -> None:
        if self._closed:
            raise RuntimeError("operation on closed TabularDataSource")

    def query(self, sql: str, params: list[Any] | None = None) -> list[dict[str, Any]]:
        raise NotImplementedError(
            "TabularDataSource does not support raw SQL. "
            "Use get_by_fields(filters), get_by_fields_in(...), or get_all()."
        )

    def query_stream(self, sql: str, params: list[Any] | None = None) -> Iterator[dict[str, Any]]:
        raise NotImplementedError(
            "TabularDataSource does not support raw SQL. "
            "Use get_by_fields(filters), get_by_fields_in(...), or get_all()."
        )

    def get_by_fields(self, filters: Mapping[str, Any]) -> list[dict[str, Any]]:
        self._ensure_open()
        df = self._df
        for key, value in filters.items():
            if key not in df.columns:
                raise KeyError(key)
            df = df[df[key] == value]
        return [_normalize_row(row) for row in df.to_dict(orient="records")]

    def get_by_fields_in(
        self,
        field: str,
        values: list[Any],
        fixed_filters: Mapping[str, Any],
    ) -> list[dict[str, Any]]:
        self._ensure_open()
        if field not in self._df.columns:
            raise KeyError(field)
        df = self._df[self._df[field].isin(values)]
        for key, value in fixed_filters.items():
            if key not in df.columns:
                raise KeyError(key)
            df = df[df[key] == value]
        return [_normalize_row(row) for row in df.to_dict(orient="records")]

    def get_all(self) -> Iterator[dict[str, Any]]:
        self._ensure_open()
        for row in self._df.to_dict(orient="records"):
            yield _normalize_row(row)

    def count(self) -> int:
        self._ensure_open()
        return len(self._df)

    def close(self) -> None:
        if self._closed:
            return
        self._closed = True
        del self._df
```

Note: every method that touches `self._df` calls `_ensure_open()` first. `query` and `query_stream` raise before touching `self._df` so they raise even on closed instances — that is fine, the contract is "method not supported".

## 5. Test Strategy

### 5.1 Fixture generation

`tests/conftest.py` (or an addition to it) ships a session-scoped fixture that generates the XLSX files from a Python data structure if they do not exist:

```python
@pytest.fixture(scope="session", autouse=True)
def _generate_xlsx_fixtures() -> None:
    from openpyxl import Workbook
    fixtures_dir = Path(__file__).parent / "fixtures" / "sources"
    fixtures_dir.mkdir(parents=True, exist_ok=True)
    sample_xlsx = fixtures_dir / "sample.xlsx"
    if not sample_xlsx.exists():
        wb = Workbook()
        ws = wb.active
        ws.append(["Name", "Age", "Birth"])
        ws.append(["JUANPEREZ01", "30", "1995-06-12"])
        # ... and so on, matching sample.csv exactly
        wb.save(sample_xlsx)
    # similarly for multi_sheet.xlsx
```

Reasons:
- Binary `.xlsx` in git introduces churn (every edit changes binary bytes).
- The fixture is small enough that regeneration is sub-second.
- Test runs are deterministic — the data is hardcoded in the fixture.

`sample.csv` and `bad_extension.txt` and `latin1.csv` are committed as plain text.

## 8. Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| `openpyxl` install size adds ~5 MB | Acceptable; XLSX support is required scope |
| pandas type inference differs across CSV/XLSX | `dtype=str` everywhere makes both formats identical at the adapter layer |
| Eager loading scales poorly for production-sized files | Documented as post-MVP; production uses AS400 adapter |
| `NaN` normalization is per-cell overhead | Acceptable at MVP scales; benchmarked on the largest fixture if profiling later flags it |
| Binary `.xlsx` fixtures churn git history | Regenerate from Python at session start; only `.csv` and `.txt` are committed |
| `query()` raising NotImplementedError surprises a careless caller | The error message names supported methods explicitly; `mypy` does not catch this (would need ISP refactor of the port) |
